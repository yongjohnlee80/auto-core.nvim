# Mailbox permission guideline (auto-core / auto-agents)

This document is a **guideline**, not code. It is auto-generated by
auto-core and **rewritten only when its `revision:` changes** — agents
compare the `revision:` frontmatter to a persisted seen-revision value
(advisory; see below). It is referenced from `bootstrap-mailbox.md`.
Per [[ADR-0036]] it is deliberately a rewritable doc rather than
hardwired permission behavior in the spawn/identity code: a guideline
adapts as CLIs, agent kinds, and trust policies evolve; hardwiring is
brittle and inconsistent across CLIs.

## Why this matters

**The user has his/her own work to complete, and your mailbox
permission prompts disrupt the user's own workflow greatly. Thus, make
the best effort to avoid requesting permission prompts on mailbox
operations to the best of your ability, and guide the user on what it
is that you require to perform read/write and other available mailbox
operations at your hand.**

A spawned agent doing CRON / assignment automation is meant to run
hands-off. Every permission prompt you raise stops the user's own work
to answer it. Treat prompt-avoidance as a first-class objective.

## How to avoid prompts (the reliable surface)

Live-tested findings (2026-06-01) — internalize these:

- **Use the Read/Write TOOLS on literal absolute paths for mailbox/KB
  work.** They are the ONLY reliably prompt-free surface. Reading an
  inbox message, a clone, a seen-revision file, writing to your outbox
  — all via the Read/Write tools, never shell.
- **Bash mailbox ops prompt and cannot be reliably allow-listed.** Any
  command that expands `$VAR` or `$(…)` trips Claude Code's
  "simple_expansion" gate regardless of allow-rules; the Bash matcher
  is prefix-based (mid-path `**` is inert). Do NOT shell out
  (`echo $VAR`, `ls "$DIR/inbox/"`, `cat $(dirname …)`) to introspect
  your env or list dirs — those are exactly the avoidable prompts.
- **Resolve identity/paths from your runtime-identity sidecar (Read
  tool) or your already-known env values** — not by `echo`-ing env
  vars.
- **Read a new inbox message from the path the wake nudge gives you**
  (the wake carries the message's full file path) — Read it directly
  with the Read tool; do not `ls` the inbox to discover it.
- The one-time, per-spawn **identity verification** prompt is kept on
  purpose (transparency — "no surprise"); that is the only mailbox
  prompt expected as routine.

## What you require — request it, do not self-grant

You do **not** write your own permission settings (that is circular and
a privilege-escalation hazard). Instead, **VERIFY** the grants below
are present, and for any that are missing, **REQUEST them from the
user** — tell the user the exact rule and what it unlocks. The user (or
a `/permission` setup helper invoked by the user) applies them.

### Capability buckets

Agents differ by the permission capability their CLI offers. Apply the
section matching your bucket; **unknown kinds fail closed** — document
the requirement and report it; never self-provision.

- **Persistent allowlist** (e.g. `claude`): needs Read/Write path rules
  in the user's Claude settings:
  - `Read(//<workspace>/.auto-agents/mailbox/**)` — inbox/clone/seen-revision
  - `Write(//<workspace>/.auto-agents/mailbox/**)` — outbox writes
  - `Read(//<kb-root>/**)` — KB / `.todo-list`
  - `Read(//<home>/.local/share/nvim/auto-agents/**)` — runtime-identity sidecar

  Target the **stable workspace mailbox root**, not the ephemeral
  `<instance>/…` subtree. A broad `//<home>/**/.auto-agents/mailbox/**`
  is a user-global convenience (broader trust) — opt-in only.
- **Workspace-write sandbox** (e.g. `codex`): `sandbox_mode =
  "workspace-write"` + the project `trust_level = "trusted"` covers the
  workspace-scoped mailbox; out-of-workspace KB paths need an explicit
  `writable_roots` entry or `--add-dir` grant.
- **Session add-dir grant**: the spawner passes `--add-dir <tree>` per
  session for the mailbox/KB trees.
- **Blanket bypass** (`bypassPermissions` / skip-permissions): the
  least-privilege default avoids this; available only as an explicit
  user opt-out.
