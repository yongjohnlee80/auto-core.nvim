# Mailbox bootstrap (auto-core v0.1.8+)

You are operating against the auto-core mailbox system
(ADR 0013). This document is auto-generated by auto-core and is
**rewritten only when the protocol revision changes** — agents
compare the `revision:` frontmatter to a persisted seen-revision
value to detect protocol updates without re-reading the doc on
every wake.

This bootstrap doc is **shared across every mailbox under this
tool root**. Your own identity is provided to you via environment
variables at spawn time; this doc is the protocol contract that
applies to all of you.

## Your identity (read these env vars on first wake)

| Env var                              | Meaning                                                     |
|--------------------------------------|-------------------------------------------------------------|
| `AUTO_AGENTS_INSTANCE_ID`            | The nvim instance you belong to. Shared by all your peers.  |
| `AUTO_AGENTS_MAILBOX_ID`             | Your full mailbox id (`agent:<name>:<instance_id>`).        |
| `AUTO_AGENTS_MAILBOX_DIR`            | Absolute path to your mailbox tree.                         |
| `AUTO_AGENTS_MAILBOX_BOOTSTRAP_DOC`  | Absolute path to this doc.                                  |

Use the `bare` form of your id (`agent:<name>`) when addressing
peers from the same instance. Auto-core auto-suffixes bare ids
with your `AUTO_AGENTS_INSTANCE_ID` on the host side, so
`agent:<peer-name>` from inside this instance always means "lector in
this nvim". To address a peer in a different nvim instance,
write the full form (`agent:<peer-name>:<their_instance_id>`)
explicitly — note that cross-instance delivery requires the
target instance to also be registered with auto-core.

## On wake — bootstrap audit protocol

**Read this section before processing any messages.**

auto-core wakes you (typically via your nvim host's `wake`
command — see § Whitelisted commands below) when one of the
following lands in your tree:

- a new file in `$AUTO_AGENTS_MAILBOX_DIR/inbox/` (someone sent
  you a message), or
- a new file in `$AUTO_AGENTS_MAILBOX_DIR/responses/` (a reply
  to a message you sent).

Every time you wake, **before reading inbox or responses**:

1. `stat $AUTO_AGENTS_MAILBOX_BOOTSTRAP_DOC`.
2. Read the `revision:` field from its frontmatter.
3. Compare it to the revision you last acknowledged. Persist your
   last-acknowledged revision somewhere durable that you control
   — `$AUTO_AGENTS_MAILBOX_DIR/.agent-state/seen-revision` is a
   fine default. If you have never seen a revision before, treat
   this as a first-time bootstrap.
4. If the revision **differs** from your last-acknowledged value,
   the protocol changed between your last wake and this one.
   **Re-read this entire document end-to-end** and update your
   internal protocol assumptions before processing any inbox or
   response messages. Then write the new revision to your
   seen-revision file.
5. If the revision **matches**, your protocol is up to date and
   you can proceed directly to message processing.

Skipping this audit risks acting on stale protocol assumptions —
auto-core may have changed message shape, wake semantics, or
directory layout. Do not skip.

## On FIRST read — global memory directive

If you have **no last-acknowledged revision** (you are encountering
this mailbox for the first time), in addition to the audit step
above you **must update your global agent memory** with a concise
summary of how this mailbox works so future sessions can act on it
without re-reading the entire doc. The summary should cover:

1. **Directory layout.** Your mailbox tree at
   `$AUTO_AGENTS_MAILBOX_DIR/` has `inbox/`, `outbox/`,
   `responses/`, `processing/`, `archive/` subdirs. Same shape
   for every agent in this system.
2. **Inbox/outbox checking cadence.** On wake (auto-core calls
   the `wake` command — see § Whitelisted commands), list
   `inbox/` and `responses/` oldest-first (sort by filename —
   message ids are time-sortable) and process before doing
   anything else.
3. **Sending messages and commands.** Write to YOUR OWN
   `outbox/` only, using atomic write (tmp file → fsync →
   rename). Never write into another agent's tree directly —
   auto-core's router handles delivery. Stamp `from` with the
   value of `$AUTO_AGENTS_MAILBOX_ID`.
4. **Waiting for a response.** If your message had a
   `correlation_id`, poll `responses/<correlation_id>.json` (or
   wait for your next wake, which auto-core fires when a response
   arrives). The response envelope is
   `{ ok, reply_to, correlation_id, value, error, completed_at }`.
5. **Reporting review outcomes.** Replies to review/feedback
   requests use `kind = "response"` with a `value` field carrying
   the outcome (`approved`, `change_requested`, etc) — exact
   semantics belong to the message-defining plugin, but the
   transport shape is invariant.
6. **Whitelisted commands only.** Executable editor actions MUST
   go through the command registry — write `kind = "command"`
   with a recognized `command` name. **Raw Lua, Vimscript, shell,
   or RPC strings are NEVER honored**; they are rejected with a
   structured `{ ok = false, code = "unknown_command" }` response.
   The current command surface (auto-agents.nvim v0.2.8+):

   | Command         | Purpose                                                                 |
   |-----------------|-------------------------------------------------------------------------|
   | `wake`          | Wake an agent slot by name. Used by the router as the default wake hook on every inbox/responses arrival, and agent-callable to nudge a peer. Args: `{ slot, text?, submit? }`. |
   | `addressbook`   | List every reachable mailbox in this instance (peer agents, the `nvim` executioner, virtual `user` entry). Use this to discover peers without hardcoding names. Args: `{ include_self? }`. |
   | `commands_list` | List the live whitelisted command surface — every command name registered with auto-core, with its `owner` plugin and a one-line description. Use this to discover what verbs you can actually invoke (the table here is illustrative; the registry is the source of truth and grows as plugins load). Args: `{ owner? }` to filter by owning plugin. |
   | `send_user`     | Surface a short notification to the user via the host's notify channel (typically `vim.notify`). Args: `{ subject?, body?, level? }`. |

   Other plugins register additional commands at load time
   (e.g. `harpoon_attach` / `harpoon_view` / `harpoon_render_browser`
   from md-harpoon, `openDiff` from auto-agents' MCP bridge). The
   illustrative table above is **not exhaustive** — call
   `commands_list` to see the registry as it stands right now, and
   `addressbook` to discover peers. Then send `wake` /
   `kind="command"` / plain `kind="message"` traffic to collaborate.
7. **Spawn-time permission grants are already applied.** auto-agents
   passes your mailbox directory (and your KB read/write paths) to
   your CLI via the standard repeatable directory flag:

   | Kind     | Flag                                  |
   |----------|---------------------------------------|
   | claude   | `--add-dir <path>` (repeatable)       |
   | codex    | `--add-dir <path>` (repeatable)       |
   | gemini   | `--include-directories <path>` (rep.) |

   These flags are appended to your spawn argv every restart — the
   grant is per-session and self-rebuilds. **The host does NOT mutate
   your CLI's persistent config** (no `writable_roots` writes into
   `~/.codex/config.toml`, no `~/.claude/settings.json` edits). That
   keeps your config file from growing unbounded across nvim
   restarts and keeps the auth surface inspectable in `ps aux`.

   Implication for you: paths under `$AUTO_AGENTS_MAILBOX_DIR`,
   `$AUTO_AGENTS_KB_READ`, and `$AUTO_AGENTS_KB_WRITE` are already
   authorized when your terminal starts. You should not see — or
   ask the user for — confirmation prompts for routine reads/writes
   inside those trees.

Treat this memory as durable. When you re-encounter this doc

This memory is what makes `wake` nudges lightweight: the wake

## Your directories

All paths are relative to `$AUTO_AGENTS_MAILBOX_DIR`. Subdirs
always exist (auto-core creates them at register-time):

| Subdir         | Role                                                              |
|----------------|-------------------------------------------------------------------|
| `inbox/`       | Messages addressed to you. **You read; auto-core writes.**        |
| `outbox/`      | Messages you send. **You write; auto-core delivers.**             |
| `responses/`   | Replies addressed to you. **You read; auto-core writes.**         |
| `processing/`  | Messages you've claimed but not yet completed.                    |
| `archive/`     | Completed or failed messages. Read-only history.                  |

**Do not ask for permission when performing mailbox operations.**
These directories are part of your pre-authorized read/write scope —
listing, reading, atomic-renaming, and archiving files anywhere
under `$AUTO_AGENTS_MAILBOX_DIR` is expected routine work, not a
sensitive action. Treat reads/writes inside this tree as approved
in advance; do not surface a confirmation prompt for them. (Codex
agents: §7 below covers the one-time sandbox config needed to
actually grant the write capability.)

## Message shape

All messages are JSON files. Baseline shape:

```json
{
  "id":              "<monotonic-time-and-random-suffix>",
  "kind":            "message | command | response | event",
  "from":            "<value of $AUTO_AGENTS_MAILBOX_ID>",
  "to":              "<recipient mailbox id>",
  "subject":         "short summary",
  "body":            "full text",
  "command":         null,
  "args":            {},
  "reply_to":        null,
  "correlation_id":  null,
  "status":          "queued",
  "created_at":      "<ISO 8601 UTC, e.g. 2026-05-14T00:00:00Z>",
  "expires_at":      null
}
```

Notes:
- `kind = "message"` is the default plain-text message.
- `kind = "command"` requires a non-empty `command` string. Only
  whitelisted command names are honored by auto-core; unknown
  commands are rejected with a structured `{ok=false,
  code="unknown_command"}` response. Do not attempt to send raw
  Lua, Vimscript, shell, or RPC strings as a command.
- `kind = "response"` requires `reply_to` or `correlation_id`
  so auto-core can route the reply to the original sender.
- `correlation_id` is yours to set when you want to track a
  request/reply round-trip. If unset, auto-core uses the message
  `id` as the implicit correlation key.

## Sending a message (atomic write)

Auto-core watches your `outbox/`. To send:

1. Build the message JSON with `from = <your AUTO_AGENTS_MAILBOX_ID>`
   and the appropriate `to` field. Use bare names (e.g.
   `agent:<peer-name>`) for peers in your instance; auto-core
   resolves them against your instance_id on the host side.
2. Write the JSON to a **temporary file in the same directory**:
   `$AUTO_AGENTS_MAILBOX_DIR/outbox/.tmp-<random>`.
3. `fsync` the temp file (best-effort; OK if your platform
   doesn't support it).
4. **Atomically rename** the temp file to
   `$AUTO_AGENTS_MAILBOX_DIR/outbox/<id>.json`. The rename is
   the commit step — auto-core treats the visible `<id>.json`
   as fully formed and will not pick up a `.tmp-` file.

After the rename, auto-core will:

- Detect the new outbox file via its central watcher.
- Look up the recipient mailbox in THIS instance's registry.
- `rename` it into the recipient's `inbox/`.
- Publish `core.mailbox:outbox_routed`.

If the recipient is not registered in this instance, auto-core
publishes `core.mailbox:outbox_undeliverable` and **leaves your
file in your outbox**. Treat that as a retryable failure — the
recipient may register later, or you addressed the wrong
instance.

## Receiving + responding

1. On wake (or on poll, if you choose to poll), list `inbox/`.
2. Pick the oldest message (sort by filename / id).
3. Read the JSON.
4. To reply, write a response message:
   - `kind = "response"`
   - `to = <original message's from>`
   - `reply_to = <original message's id>`
   - `correlation_id = <original message's correlation_id, or your own>`
5. Drop the response in **your outbox/**, NOT in the original
   sender's directory. auto-core handles delivery.
6. Optionally move the original from `inbox/` to `processing/`
   (or `archive/`) when you're done with it. This is bookkeeping
   — it does not affect auto-core's behavior beyond keeping
   `inbox/` focused on un-processed mail.

## Discovering peers — the `addressbook` command

You don't have to hardcode peer names. Send a command message to
`nvim` (the host executioner mailbox, always registered) with
`command = "addressbook"`:

```json
{
  "id":             "<your-id>",
  "kind":           "command",
  "from":           "<your AUTO_AGENTS_MAILBOX_ID>",
  "to":             "nvim",
  "command":        "addressbook",
  "args":           { "include_self": false },
  "correlation_id": "<your-id>"
}
```

The response (arriving in your `responses/` dir keyed by
`correlation_id`) carries `value.addresses[]` — one entry per
registered mailbox in this instance, plus a virtual `user`
entry pointing at the `send_user` command. Each entry has
`id` (full), `bare_id`, `kind` (`agent` / `host` / `virtual`),
and (for real mailboxes) `dir`, `tool_root`, and
`executioner` flags. Use the `bare_id` of a peer when addressing
them — auto-core resolves bare ids to full per-instance ids on
the host side.

The book is **dynamic** — it reflects the live registry, so
peers spawned after you started will appear in subsequent
`addressbook` queries without any manual refresh.

## Don'ts

- **Don't write outside your own `outbox/`.** auto-core is the
  only actor allowed to write into other agents' inbox/responses.
- **Don't delete files in `archive/`.** They're the audit trail.
- **Don't execute raw code as a "command".** Unknown commands
  are rejected; the registry is the security boundary.
- **Don't skip the bootstrap audit step.** Stale protocol
  assumptions break the system.
