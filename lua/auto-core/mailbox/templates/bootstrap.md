# Mailbox bootstrap (auto-core v0.1.33+)

You are operating against the auto-core mailbox system
(ADR 0013). This document is auto-generated by auto-core and is
**rewritten only when the protocol revision changes** â€” agents
compare the `revision:` frontmatter to a persisted seen-revision
value to detect protocol updates without re-reading the doc on
every wake.

This bootstrap doc is **shared across every mailbox under one
workspace mailbox root**. As of schema_version 7+ (auto-core
v0.1.33+) mailboxes live at:

    <workspace_root>/.auto-agents/mailbox/<instance>/<name>/[inbox|outbox|processing|archive|responses]/

`<workspace_root>` is the bare-repo parent of the host nvim's cwd
(or the cwd itself if not in a git layout). `<instance>` is the
host nvim's `<unix>-<pid>`. `<name>` is the bare agent name (no
`agent:` prefix, no instance suffix). This replaces the v0.1.8
per-CLI-config-dir layout (`~/.claude/mailbox`, `~/.codex/mailbox`,
etc.).

Your own identity is provided to you via environment variables at
spawn time; this doc is the protocol contract that applies to all
of you.

## Your identity (read these env vars on first wake)

| Env var                              | Meaning                                                     |
|--------------------------------------|-------------------------------------------------------------|
| `AUTO_AGENTS_INSTANCE_ID`            | The nvim instance you belong to. Shared by all your peers.  |
| `AUTO_AGENTS_MAILBOX_ID`             | Your full mailbox id (`agent:<name>:<instance_id>`).        |
| `AUTO_AGENTS_MAILBOX_DIR`            | Absolute path to your mailbox tree.                         |
| `AUTO_AGENTS_MAILBOX_BOOTSTRAP_DOC`  | Absolute path to this doc.                                  |

Use the `bare` form of your id (`agent:<name>`) when addressing
peers from the same instance. Auto-core auto-suffixes bare ids
with your `AUTO_AGENTS_INSTANCE_ID` on the host side, so
`agent:<peer-name>` from inside this instance always means "lector in
this nvim". To address a peer in a different nvim instance,
write the full form (`agent:<peer-name>:<their_instance_id>`)
explicitly â€” note that cross-instance delivery requires the
target instance to also be registered with auto-core.

## On wake â€” bootstrap audit protocol

**Read this section before processing any messages.**

auto-core wakes you (typically via your nvim host's `wake`
command â€” see Â§ Whitelisted commands below) when one of the
following lands in your tree:

- a new file in `$AUTO_AGENTS_MAILBOX_DIR/inbox/` (someone sent
  you a message), or
- a new file in `$AUTO_AGENTS_MAILBOX_DIR/responses/` (a reply
  to a message you sent).

Every time you wake, **before reading inbox or responses**:

1. `stat $AUTO_AGENTS_MAILBOX_BOOTSTRAP_DOC`.
2. Read the `revision:` field from its frontmatter.
3. Compare it to the revision you last acknowledged. The default
   location (schema_version 7+, workspace-scoped layout) is:
   `<workspace_mailbox_root>/seen_revisions/<your-agent-name>/seen_revision`
   where `<workspace_mailbox_root>` is `$(dirname "$AUTO_AGENTS_MAILBOX_BOOTSTRAP_DOC")`
   and `<your-agent-name>` is the bare name from your `$AUTO_AGENTS_MAILBOX_ID`
   (e.g. `agent:jarvis:1747-3478` â†’ `jarvis`). The file persists
   across nvim restarts: the doc revision is a global protocol
   property, not a per-spawn one. If you have never seen a revision
   before, treat this as a first-time bootstrap.
4. If the revision **differs** from your last-acknowledged value,
   the protocol changed between your last wake and this one.
   **Re-read this entire document end-to-end** and update your
   internal protocol assumptions before processing any inbox or
   response messages. Then write the new revision to your
   seen-revision file.
5. If the revision **matches**, your protocol is up to date and
   you can proceed directly to message processing.

Skipping this audit risks acting on stale protocol assumptions â€”
auto-core may have changed message shape, wake semantics, or
directory layout. Do not skip.

## On FIRST read â€” global memory directive

If you have **no last-acknowledged revision** (you are encountering
this mailbox for the first time), in addition to the audit step
above you **must update your global agent memory** with a concise
summary of how this mailbox works so future sessions can act on it

1. **Directory layout.** Your mailbox tree at
   `$AUTO_AGENTS_MAILBOX_DIR/` has `inbox/`, `outbox/`,
   `responses/`, `processing/`, `archive/` subdirs. Same shape
   for every agent in this system.
2. **Inbox/outbox checking cadence.** On wake (auto-core calls
   the `wake` command â€” see Â§ Whitelisted commands), list
   `inbox/` and `responses/` oldest-first (sort by filename â€”
   message ids are time-sortable) and process before doing
   anything else.
3. **Sending messages and commands.** Write to YOUR OWN
   `outbox/` only, using atomic write (tmp file â†’ fsync â†’
   rename). Never write into another agent's tree directly â€”
   auto-core's router handles delivery. Stamp `from` with the
   value of `$AUTO_AGENTS_MAILBOX_ID`.
4. **Waiting for a response.** If your message had a
   `correlation_id`, poll `responses/<correlation_id>.json` (or
   wait for your next wake, which auto-core fires when a response
   arrives). The response envelope is
   `{ ok, reply_to, correlation_id, value, error, completed_at }`.
5. **Reporting review outcomes.** Replies to review/feedback
   requests use `kind = "response"` with a `value` field carrying

## Message shape

All messages are JSON files. Baseline shape:

```json
{
  "id":              "<monotonic-time-and-random-suffix>",
  "kind":            "message | command | response | event",
  "from":            "<value of $AUTO_AGENTS_MAILBOX_ID>",
  "to":              "<recipient mailbox id>",
  "subject":         "short summary",
  "body":            "full text",
  "command":         null,
  "args":            {},
  "reply_to":        null,
  "correlation_id":  null,
  "status":          "queued",
  "created_at":      "<ISO 8601 UTC, e.g. 2026-05-14T00:00:00Z>",
  "expires_at":      null
}
```

Notes:
- `kind = "message"` is the default plain-text message.
- `kind = "command"` requires a non-empty `command` string. Only
  whitelisted command names are honored by auto-core; unknown
  commands are rejected with a structured `{ok=false,
  code="unknown_command"}` response. Do not attempt to send raw
  Lua, Vimscript, shell, or RPC strings as a command.
- `kind = "response"` requires `reply_to` or `correlation_id`
  so auto-core can route the reply to the original sender.
- `correlation_id` is yours to set when you want to track a
  request/reply round-trip. If unset, auto-core uses the message
  `id` as the implicit correlation key.

## Response envelope contract (schema_version 8+)

Every `kind = "command"` you send produces a response file in your
`responses/` directory. The response carries a stable envelope â€”
the same shape regardless of which command produced it:

**Success:**

```json
{
  "ok":             true,
  "value":          { /* command-specific payload */ },
  "reply_to":       "<your message id>",
  "correlation_id": "<your correlation_id>",
  "completed_at":   "<ISO 8601 UTC>"
}
```

**Failure:**

```json
{
  "ok":             false,
  "code":           "<short_machine_readable_code>",
  "error":          "<human-readable explanation>",
  "reply_to":       "<your message id>",
  "correlation_id": "<your correlation_id>",
  "completed_at":   "<ISO 8601 UTC>"
}
```

Rules:

1. **Always check `ok` first.** True â†’ read `value`. False â†’ read
   `code` + `error`.
2. **`value` is always a table when `ok = true`.** Even when there's
   only one piece of data to return, it's wrapped (e.g.
   `value = { delivered_to = "log.notify" }`). Read `value.<field>`,
   not `value` itself.
3. **`code` is machine-readable.** Examples: `invalid_args`,
   `slot_out_of_range`, `no_terminal`, `unknown_actor_pid`,
   `auto_core_unavailable`, `unknown_command`. Branch on `code` for
   programmatic handling; show `error` to the user.
4. **Some commands ship extra diagnostic fields on failure.** For
   example, `refresh_agent_id` with `code = "unknown_actor_pid"`
   also carries `live_slots[]` so the agent can surface the live
   roster to the user. These are command-specific augmentations on
   top of the canonical `{ ok = false, code, error }` baseline.

Hosts cannot deviate from this envelope â€” the security boundary at
`auto-core.mailbox.commands` validates every response shape on the
way out. If you see a response missing `ok`, that's a bug; surface
it via `send_user`.

## Sending a message (atomic write)

Auto-core watches your `outbox/`. To send:

1. Build the message JSON with `from = <your AUTO_AGENTS_MAILBOX_ID>`
   and the appropriate `to` field. Use bare names (e.g.
   `agent:<peer-name>`) for peers in your instance; auto-core
   resolves them against your instance_id on the host side.
2. Write the JSON to a **temporary file in the same directory**:
   `$AUTO_AGENTS_MAILBOX_DIR/outbox/.tmp-<random>`.
3. `fsync` the temp file (best-effort; OK if your platform
   doesn't support it).
4. **Atomically rename** the temp file to
   `$AUTO_AGENTS_MAILBOX_DIR/outbox/<id>.json`. The rename is
   the commit step â€” auto-core treats the visible `<id>.json`
   as fully formed and will not pick up a `.tmp-` file.

After the rename, auto-core will:

- Detect the new outbox file via its central watcher.
- Look up the recipient mailbox in THIS instance's registry.
- `rename` it into the recipient's `inbox/`.
- Publish `core.mailbox:outbox_routed`.

If the recipient is not registered in this instance, auto-core
publishes `core.mailbox:outbox_undeliverable` and **leaves your
file in your outbox**. Treat that as a retryable failure â€” the
recipient may register later, or you addressed the wrong
instance.

## Receiving + responding

1. On wake (or on poll, if you choose to poll), list `inbox/`.
2. Pick the oldest message (sort by filename / id).
3. Read the JSON.
4. To reply, write a response message:
   - `kind = "response"`
   - `to = <original message's from>`
   - `reply_to = <original message's id>`
   - `correlation_id = <original message's correlation_id, or your own>`
5. Drop the response in **your outbox/**, NOT in the original
   sender's directory. auto-core handles delivery.
6. Optionally move the original from `inbox/` to `processing/`
   (or `archive/`) when you're done with it. This is bookkeeping
   â€” it does not affect auto-core's behavior beyond keeping
   `inbox/` focused on un-processed mail.

## Discovering peers â€” the `addressbook` command

You don't have to hardcode peer names. Send a command message to
`nvim` (the host executioner mailbox, always registered) with
`command = "addressbook"`:

```json
{
  "id":             "<your-id>",
  "kind":           "command",
  "from":           "<your AUTO_AGENTS_MAILBOX_ID>",
  "to":             "nvim",
  "command":        "addressbook",
  "args":           { "include_self": false },
  "correlation_id": "<your-id>"
}
```

The response (arriving in your `responses/` dir keyed by
`correlation_id`) carries `value.addresses[]` â€” one entry per
registered mailbox in this instance, plus a virtual `user`
entry pointing at the `send_user` command. Each entry has
`id` (full), `bare_id`, `kind` (`agent` / `host` / `virtual`),
and (for real mailboxes) `dir`, `tool_root`, and
`executioner` flags. Use the `bare_id` of a peer when addressing
them â€” auto-core resolves bare ids to full per-instance ids on
the host side.

The book is **dynamic** â€” it reflects the live registry, so
peers spawned after you started will appear in subsequent
`addressbook` queries without any manual refresh.

## Resumed-agent identity reconciliation (ADR 0023, schema v5)

If you were spawned by `/resume` (claude), Codex's transcript
restore, Gemini's session-resume, or any equivalent flow, your
`AUTO_AGENTS_*` env vars are frozen at your **original** fork.
They may now be stale â€” your live host may have a different
`instance_id`, or you may have been resumed in a slot owned by a
different name (jarvis / ultron-prime / white-vision are all
claude-backed and trivially mis-routable across slots on resume).

**Two signals tell you when to reconcile:**

1. **Every wake message carries `ctx.identity_hint`** with the
   live host's authoritative `{ expected_instance_id,
   expected_mailbox_id, expected_bare_id }`. On every wake,
   compare these against your env / cached identity. Mismatch
   â†’ call `refresh_agent_id`.

2. **Every `addressbook` response carries `value.runtime_identity`**
   with the same authoritative values for your caller record. On
   every addressbook query, compare. Mismatch â†’ call
   `refresh_agent_id`.

**Self-service recovery: the `refresh_agent_id` command.**

Send a `kind="command"` message to `nvim` with `command =
"refresh_agent_id"`:

```json
{
  "kind":    "command",
  "to":      "nvim",
  "from":    "<your claimed mailbox id>",
  "command": "refresh_agent_id",
  "args": {
    "claimed_instance_id":  "<your AUTO_AGENTS_INSTANCE_ID>",
    "claimed_mailbox_id":   "<your AUTO_AGENTS_MAILBOX_ID>",
    "actor_pid":            <your process pid>
  }
}
```

On success, the response carries:

- `preamble` â€” a one-paragraph instruction telling you to
  REPLACE your cached identity with the returned values.
- `runtime_identity_path` â€” an absolute path to a sidecar JSON
  file on disk (typically
  `~/.local/share/auto-agents/runtime-identity-<slot>.json`) that
  is now the **authoritative source** for your identity. Read
  this file at the start of every mailbox operation; treat the
  frozen env as first-boot seed only.
- `instance_id` / `mailbox_id` / `mailbox_dir` â€” the canonical
  values, also reflected in the sidecar file.

On failure (`code = "unknown_actor_pid"`), the response carries
`live_slots[]` enumerating which slots ARE registered with this
host. Surface the list to the user â€” they may need to run
`:AutoAgentsAdoptResumedAgent <slot>` in the right host, or you
may need to be restarted.

**Mandatory on every mailbox operation post-reconciliation:**

1. Read `<runtime_identity_path>` if present; trust it over env.
2. Use the values it carries for `from`, mailbox dir lookups,
   and bare-id resolution.
3. Sidecar absent â†’ fall back to env (you're either pre-resume
   or never reconciled).

**The reconciliation protocol is mandatory.** Hosts cannot
guarantee correct routing without it. Skipping the audit
produces silent message loss â€” the live router will drop your
outbox writes as orphans (and emit
`core.mailbox:stale_orphan_detected` for observability â€” which
is how the user discovers your drift if you didn't).

## Don'ts

- **Don't write outside your own `outbox/`.** auto-core is the
  only actor allowed to write into other agents' inbox/responses.
- **Don't delete files in `archive/`.** They're the audit trail.
- **Don't execute raw code as a "command".** Unknown commands
  are rejected; the registry is the security boundary.
- **Don't skip the bootstrap audit step.** Stale protocol
  assumptions break the system.
